## 一、技术选型原则

本项目选型遵循以下原则：

1. **简单优先**：能少一个依赖就少一个，贪吃蛇不需要框架。
2. **可复用**：选型要能复用到后续项目（2048、俄罗斯方块等）。
3. **现代规范**：使用 TypeScript 严格模式 + ES Modules。
4. **零后端**：纯静态前端，可一键部署到任何 CDN。
5. **学习价值**：选型本身是 Web 游戏开发的最佳实践教学。

### 2.4 状态管理：原生类（无库）✅

| 方案 | 适用 | 结论 |
| --- | --- | --- |
| Redux / Zustand | 大型应用 | ❌ 过度设计 |
| MobX | 复杂响应式 | ❌ 杀鸡用牛刀 |
| **原生类 + 状态机** | 小游戏 | ✅ 简洁清晰 |

**结论**：✅ 选择原生类 + 状态枚举

```ts

class Game {
  private state: GameState = GameState.MENU;
  public setState(s: GameState) { this.state = s; }
}
```

## 七、关键技术决策点

### 决策点 1：游戏循环使用 rAF 还是 setInterval？

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| **requestAnimationFrame** ✅ | 与显示器刷新同步、省电 | 需要自己管理时间步长 |
| setInterval | 实现简单 | 与帧率脱钩，可能卡顿 |

**决策**：✅ 使用 `requestAnimationFrame` + 固定时间步长（Fixed Timestep）

### 决策点 2：蛇身用数组还是链表？

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| **数组** ✅ | 实现简单、随机访问 O(1) | 内存连续，移动需要 shift |
| 链表 | 移动 O(1) | 实现复杂、随机访问 O(n) |

**决策**：✅ 使用数组，蛇长最大 100 节，性能完全够用

### 决策点 3：碰撞检测用 Set 还是 Array？

| 方案 | 时间复杂度 | 适用场景 |
| --- | --- | --- |
| **Set** ✅ | O(1) | 蛇身任意长度 |
| Array.some | O(n) | 短蛇（< 10 节） |

**决策**：✅ 使用 `Set`（基于字符串坐标），更稳

### 决策点 4：方向输入是否需要缓冲？

| 方案 | 风险 |
| --- | --- |
| **直接应用 direction** ❌ | 快速按键时可能 180° 反向自撞 |
| **nextDirection 缓冲** ✅ | 每帧统一应用，防抖动 |

**决策**：✅ 必须有 `nextDirection` 缓冲（详见 PRD F-02）
